macOS APP 簽名

發表於 2025-01-07 分類於 AppleDeveloper

所有上傳到App Store的App都會透過Xcode做簽名與公證，所以開發者無需另外處理。
如果不要透過App Store分發App則需要做簽名與公證，否則使用者都無法安裝APP。

簽名

一般來說，一個專案簽一次就好，但是如果專案裡面有其他的執行檔，如.sh，.app等等，這些執行檔就要做額外做簽名。

如果專案裡面，有三個.sh，就必須將三個.sh額外做簽名，匯出.app之後要再做一次，爾後，如果有修改過內容，也要重新簽名。

目前來說，簽名都要用同一個帳號做簽章。

指令範例：

codesign --keychain "/Library/Keychains/keychain" --sign "Developer ID Application: RRRR TECH CO., LTD (YYJJNNNBYG)" --timestamp --options runtime --verbose=4 /Users/Shared/App

說明：

–keychain “/Library/Keychains/keychain”

  • 指定使用的鑰匙串 (Keychain) 來查找簽名所需的憑證。
  • 在這裡，使用的是系統鑰匙串 /Library/Keychains/System.keychain，這樣可以確保所需的憑證從系統範圍的 Keychain 中找到。

–sign “Developer ID Application: RRRR TECH CO., LTD (YYJJNNNBYG)”

  • 用於指定簽名憑證。
  • 這裡使用的是「Developer ID Application」的憑證，這類憑證允許將應用分發給 macOS 使用者，並避免 Gatekeeper 進行警告。
  • RRRR TECH CO., LTD (YYJJNNNBYG) 是這個憑證的名稱和團隊 ID，可以確保簽名是由授權的開發者完成的。

–timestamp

  • 此參數會將時間戳記加到簽名中，以確保簽名的持久有效性。
  • 即使憑證過期，有時間戳記的簽名依然會被視為有效，適合長期分發的應用。

–options runtime

  • 開啟 Apple 的「執行時安全性」功能，這包括系統完整性保護 (SIP) 和代碼完整性檢查。
  • 這樣可以增加應用的安全性，防止被修改或植入惡意程式碼。這個選項在 macOS Mojave 及之後的版本是必要的。

–verbose=4

  • 設置詳情輸出等級，數字 4 代表最高的詳細級別，會提供更多的輸出訊息，幫助在簽名過程中進行故障排查。

/Users/Shared/App

  • 這是需要進行簽名的目標檔案或應用的路徑。
  • 此目錄下的 App 是進行簽名的主體，命令會遞歸地對該應用及其所有包含的可執行檔案進行簽名。

查看簽名內容

在公證的時候，如果常常無法成功，就查看簽名的內容，看簽名跟公證用的帳號是不是同一個。

下圖為簽名內容

指令範例：

codesign -dvvv /path/to/xxx.app

路徑要寫絕對路徑。

重新簽名

如果發現簽名的ID是錯的，就要重新簽。

指令範例：

codesign --force --deep --keychain "/Library/Keychains/System.keychain" --sign "Developer ID Application: AAAA INC. (5Q55QYY5AA)" --timestamp --options runtime --verbose=4 /Users/onAndDecryption1/App
